# PromptAIChemy — Project Reference

## Overview

**PromptAIChemy** (`CodeWithAkhil.promptaichemy`) is a VS Code / Cursor extension that rewrites rough developer instructions into clear, specific, context-aware prompts for AI coding assistants (Claude Code, Cursor, Copilot, etc.). It uses a **locally running Ollama model** — no data leaves the machine.

The workflow:
1. Developer types a rough instruction in the sidebar.
2. The extension collects workspace context (folder tree, `CLAUDE.md`, open file).
3. Context + raw prompt are sent to Ollama via HTTP.
4. The polished prompt is displayed in the sidebar, ready to copy.

## File Descriptions

### `src/extension.ts`
Entry point called by VS Code when the extension activates. Instantiates `SidebarProvider` with `context.extensionUri` and registers it as the webview view provider for the view ID `promptEnhancer.sidebar`. Also exports a no-op `deactivate()`.

### `src/sidebarProvider.ts`
Implements `vscode.WebviewViewProvider`. Key responsibilities:

- **`resolveWebviewView()`** — sets webview options, renders HTML (with embedded `window.__INIT__` state), registers config and workspace-folder change listeners, and wires the message handler.
- **`_handleEnhance()`** — reads config fresh on every click (`getConfiguration()`), collects context via `contextReader`, calls `ollamaClient.enhancePrompt`, then posts `result` or `error` back to the webview. Catches `OllamaUnreachableError` and `OllamaModelNotFoundError` to show specific `vscode.window.showErrorMessage` toasts.
- **`_getHtmlForWebview()`** — builds the full sidebar HTML inline, including all CSS (VS Code CSS variables for theme compatibility), the DOM structure, and an inline `<script>` that seeds `window.__INIT__` with `{ model, hasWorkspace }`.
- Listeners are disposed on `webviewView.onDidDispose` to prevent leaks.

### `src/ollamaClient.ts`
Pure HTTP module (no VS Code API).

- **`OllamaUnreachableError`** — thrown when `fetch()` itself rejects (Ollama not running / wrong port).
- **`OllamaModelNotFoundError`** — thrown on HTTP 404 (model not pulled).
- **`enhancePrompt()`** — assembles a structured user message from context parts (folder tree → CLAUDE.md → open file → raw prompt), sends `POST /api/generate` with `stream: false`, and returns `data.response.trim()`.

System prompt instructs the model to act as an expert prompt engineer: rewrite for specificity, reference file names from the tree, incorporate CLAUDE.md conventions, and return only the enhanced prompt with no preamble.

### `src/contextReader.ts`
Three pure utility functions:

- **`readClaudeMd(workspaceRoot)`** — reads `CLAUDE.md` via `vscode.workspace.fs.readFile`; returns `''` if absent.
- **`readOpenFile(editor)`** — returns `editor.document.getText()` or `''` if no editor is open.
- **`readFolderTree(workspaceRoot)`** — synchronous recursive walk up to depth 2, skipping `node_modules`, `.git`, `dist`, `out`, `.vscode-test`, and hidden files/dirs. Sorts directories before files at each level. Returns a plain-text indented tree.

### `media/main.js`
Runs inside the sandboxed webview (CSP-restricted, no Node.js). Uses `acquireVsCodeApi()` to communicate with the extension host via `postMessage`.

Key behaviours:
- Reads `window.__INIT__` on load to set model label and workspace state.
- `promptInput` `input` event clears the result area immediately (so stale results don't linger).
- Disables the Enhance button and textarea when no workspace is open; shows a warning banner.
- Disables the button while a request is in flight (`inFlight` flag).
- Handles messages: `result` (show enhanced prompt), `error` (show inline hint; full message shown via VS Code notification), `updateState` (sync model label and workspace state after config change).
- Copy button briefly shows "Copied!" for 1.5 s for visual feedback.

### `package.json`
Extension manifest and npm configuration.

- **Publisher:** `CodeWithAkhil`
- **Extension ID:** `CodeWithAkhil.promptaichemy`
- **Activation:** `[]` — activates lazily when the sidebar view is first opened.
- **Contributes:**
  - `viewsContainers.activitybar` — registers the `promptEnhancer` container with a `$(edit)` icon.
  - `views.promptEnhancer` — registers `promptEnhancer.sidebar` as a webview view.
  - `configuration` — two user settings (see Configuration section).

### `tsconfig.json`
- Target: `ES2020`, module: `commonjs`
- `strict: true`, `esModuleInterop: true`, `skipLibCheck: true`
- Source root: `src/`, output: `dist/` (output only used for TypeScript type-checking; esbuild handles the actual bundle)

### `.vscodeignore`
Excludes from `.vsix`: `.vscode/`, `src/`, `node_modules/`, `.gitignore`, `tsconfig.json`, source maps, `.ts` files. Only `dist/` and `media/` ship in the package.

## Configuration Settings

| Setting | Type | Default | Description |
|---|---|---|---|
| `promptEnhancer.ollamaUrl` | `string` | `http://localhost:11434` | Base URL of the local Ollama server |
| `promptEnhancer.model` | `string` | `llama3.1` | Ollama model name to use |

Both settings are read at enhancement time (not at activation), so changes apply immediately without reloading the extension.

## Architecture

### Communication Pattern
```
Webview (media/main.js)
    │  postMessage({ command: 'enhance', prompt })
    ▼
SidebarProvider._handleEnhance()
    │  readClaudeMd / readOpenFile / readFolderTree
    │  ollamaClient.enhancePrompt()  →  POST /api/generate
    │
    ▼  postMessage({ command: 'result', text })
Webview (media/main.js)
```

### Key Design Decisions

| Decision | Rationale |
|---|---|
| esbuild bundling (not `tsc`) | Single-file output, zero runtime deps, ~14 KB bundle |
| `stream: false` for Ollama | Simpler request/response cycle; streaming adds complexity with no UX benefit in a sidebar |
| Config read at click time | Settings changes take effect without extension reload |
| `window.__INIT__` seed | Avoids timing issues between webview HTML load and first `postMessage` |
| Typed error classes | Allows `sidebarProvider` to show specific, actionable error messages per failure mode |
| Webview CSP nonce | Satisfies VS Code security requirements; blocks inline scripts except the seeded `__INIT__` block |
| `onDidDispose` cleanup | Config and workspace listeners are disposed with the view to prevent memory leaks |
